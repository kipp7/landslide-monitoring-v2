# Decision: use-reverse-tunnel-and-domain-ready-cloud-endpoint-for-rk3568

## Context

The RK3568 edge gateway now uses EC200A-CN 4G as a formal cloud uplink. The user is concerned that future cloud server replacement may scatter configuration changes across RK3568, the desktop package, MQTT, and deployment records. The user also asked whether RK3568 can directly have a public IP.

## Decision

Keep RK3568 remote access behind a reverse SSH tunnel initiated by RK3568 to the cloud server. Do not expose RK3568 SSH or sidecar ports directly to the public internet.

Treat the cloud endpoint as replaceable infrastructure. For now, the active endpoint is `134.175.187.208`; long term, prefer domain-based API/MQTT endpoints so server replacement can be handled mostly by DNS plus a small RK3568 route/tunnel update.

## Rationale

- EC200A currently receives carrier-internal addressing:
  - cellular PDP address: `10.217.14.146`
  - RK3568 `usb0`: `192.168.43.100/24`
- Public internet clients cannot directly dial into those addresses.
- Direct public SSH would require public APN, fixed public IP, VPDN/APN private network, or carrier VPN service, and would still be a security risk.
- Reverse SSH is safer and already verified:
  - `127.0.0.1:22079` -> RK3568 SSH
  - `127.0.0.1:28081` -> field-link-monitor
  - `127.0.0.1:28082` -> Hermes supervisor
  - `127.0.0.1:28087` -> alarm actuator
- Keeping tunnel ports on cloud `127.0.0.1` prevents public scanning and accidental sidecar exposure.
- Cloud Docker containers use a host-local bridge service instead of public exposure:
  - `lsmv2-docker-loopback-bridge.service`
  - reproducible installer: `scripts/deploy/install-cloud-rk3568-loopback-bridge.sh`
  - `172.18.0.1:28081/28082/28087` -> cloud host `127.0.0.1:28081/28082/28087`
  - API env uses these `172.18.0.1` URLs with `RK3568_STATUS_HTTP_TIMEOUT_MS=6000`.

## Consequences

- Future cloud server replacement must update:
  - cloud Docker/Compose env and data backup restore
  - RK3568 `/etc/lsmv2/rk3568-cellular-cloud.env`
  - RK3568 `/etc/lsmv2/field-gateway.env`
  - RK3568 `/etc/lsmv2/rk3568-reverse-tunnel.env`
  - new cloud `rk3568-tunnel` user and authorized public key
  - cloud `lsmv2-docker-loopback-bridge.service` if API containers need RK3568 sidecar access
  - cloud API env `RK3568_ALARM_ACTUATOR_URL`, `RK3568_FIELD_LINK_MONITOR_URL`, `RK3568_HERMES_EDGE_SUPERVISOR_URL`, `RK3568_STATUS_HTTP_TIMEOUT_MS`
  - Windows desktop cloud package `desk-runtime.json`, generated by `prepare-desk-win-cloud-delivery.ps1`
  - deployment docs and journal
- Future cloud server replacement should not require changing:
  - RK2206 firmware
  - XLS1/XL01 protocol
  - COBS + CRC v1 frames
  - RS485 sensor fields
  - model structure
  - desktop UI business logic

## Follow-up

- Use [docs/guides/deployment/cloud-server-migration-and-rk3568-public-access.md](/E:/学校/02%20项目/99%20山体滑坡优化完善/landslide-monitoring-v2-mainline/docs/guides/deployment/cloud-server-migration-and-rk3568-public-access.md) as the operational checklist.
- If using a domain later, update the RK3568 route guard to resolve the domain before installing a host route, or keep updating the resolved server IP in the RK3568 env.
- If productizing RK3568 sidecar status through the cloud, proxy it through authenticated cloud API instead of exposing raw tunnel ports.
