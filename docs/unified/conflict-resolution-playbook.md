# 冲突处理手册

## 1. 目标

本手册用于解决多 CLI / 多 worktree 并发开发时的冲突问题。

冲突不可避免，但必须可控、可追溯、可快速恢复。

## 2. 冲突优先级原则

始终按以下优先级处理：

1. **统一文档与当前基线**
2. **协议与共享字段**
3. **平台服务实现**
4. **客户端适配**
5. **局部样式与非关键实现细节**

也就是说：

- 文档冲突优先解决
- 契约冲突优先于页面冲突
- 共享层冲突优先于下游业务层冲突

## 3. 冲突分类

### A 类：文档冲突

例如：

- 路由说明不一致
- GNSS 字段定义不一致
- 任务边界不一致

处理方式：

- 先统一文档
- 再要求各线按统一文档调整代码

### B 类：接口与字段冲突

例如：

- `device_id` / `station_id` 语义不一致
- `/api/*` 与 `/api/v1` 响应结构不一致
- Desk 和平台字段命名不一致

处理方式：

- 先统一接口清单与字段表
- 若无法立即统一，明确兼容层

### C 类：实现冲突

例如：

- 两条线同时改了 `services/api`
- 两条线同时改了 `apps/desk`

处理方式：

- 先看哪条线依赖更上游的契约
- 优先保留符合当前统一契约的一方
- 另一方做适配或局部重做

### D 类：历史代码与新主线冲突

例如：

- 旧集成仓逻辑和新主线逻辑不同
- 参考仓旧实现和当前文档不一致

处理方式：

- 以当前长期主线仓和统一文档为准
- 参考仓只用于提供依据，不直接压过当前主线

## 4. 标准处理流程

### 第一步：先暂停直接合并

遇到冲突先不要硬合。

### 第二步：判断冲突类型

判断属于：

- 文档冲突
- 协议冲突
- 实现冲突
- 历史迁移冲突

### 第三步：找权威来源

优先顺序：

1. `docs/unified/*`
2. `docs/integrations/*`
3. `docs/architecture/*`
4. 任务队列与任务单
5. 参考仓历史资料

### 第四步：先修上游，再修下游

例如：

- 先定字段
- 再改 API
- 再改 Desk

### 第五步：只在 `integration` 收口

- 冲突先在 `integration` 处理
- 不要直接在两个专题 worktree 之间互相覆盖

## 5. 常见冲突场景与建议

### 场景 1：Desk 接口和平台接口不一致

建议：

- 先更新 `desk-api-align` 任务输出
- 再决定是 Desk 适配还是平台提供兼容层

### 场景 2：GNSS 字段和算法输入不一致

建议：

- 先统一 `gnss-protocol`
- 再要求 `algo-inventory` 按统一字段重写卡片

### 场景 3：平台恢复线与 Desk 线同时改 API

建议：

- 平台恢复线先确定真实服务现状
- Desk 线不直接猜 API，以恢复线结论为准

### 场景 4：有人改了旧集成仓

建议：

- 不把旧集成仓改动视为主线事实
- 先判断是否值得迁移到 `mainline`
- 迁移时重新整理，不直接整仓覆盖

## 6. 冲突记录要求

每次明显冲突都至少记录：

- 冲突发生在哪两个任务之间
- 涉及哪些文件或模块
- 最终以什么为准
- 哪一方需要重做或适配

可记录到：

- `docs/journal/2026-03.md`
- 或对应专题任务文档

## 7. 禁止的冲突处理方式

- 不允许“谁先提交谁算”
- 不允许跳过文档直接按个人理解硬改
- 不允许直接覆盖别人 worktree 的结果
- 不允许在 `main` 上直接处理复杂冲突

## 8. 一句话原则

先统一文档，  
再统一字段，  
最后统一代码。
