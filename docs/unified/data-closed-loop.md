# 数据闭环统一说明

## 1. 为什么先统一数据闭环

当前项目最关键的不是单点功能，而是“数据从设备产生到最终展示”的整条链。

只要这条链没有打通：

- 桌面端再漂亮也只是壳
- 算法再多也没有稳定输入
- 规则和告警也无法稳定落地

## 2. 当前统一闭环

### 第 1 段：设备采集

设备从 GNSS 与其他传感器采集数据。

输出应至少包括：

- `device_id`
- 时间
- 遥测指标
- 可选质量信息
- 固件与采样元数据

### 第 2 段：设备上报

设备按 MQTT Topic 与 TelemetryEnvelope 上报：

- `telemetry/{device_id}`
- 可选 `presence/{device_id}`
- 命令接收：`cmd/{device_id}`
- 命令回执：`cmd_ack/{device_id}`

### 第 3 段：接入与校验

ingest 负责：

- 鉴权通过后的消息接入
- Envelope 校验
- 补服务端接收时间
- 合法消息转发 Kafka
- 非法消息进入 DLQ

### 第 4 段：消息缓冲

Kafka 负责：

- 削峰
- 解耦
- 支持延迟消费
- 支持问题消息回放与追踪

### 第 5 段：遥测入库

telemetry-writer 负责：

- 读取标准化遥测
- 按 `sensor_key + value_*` 拆为稀疏点位
- 写入 ClickHouse

### 第 6 段：规则与算法

rule-engine / 算法链负责：

- 读取标准化数据或聚合结果
- 按规则、阈值、窗口与算法输出告警
- 记录 evidence
- 写入 PostgreSQL 告警事件

### 第 7 段：接口层

API 负责：

- 给客户端提供设备、站点、曲线、状态、基线、告警等查询接口
- 统一响应结构
- 提供实时最新值与告警联动能力

### 第 8 段：可视化与操作

Desk / Web 负责：

- 展示设备状态
- 展示 GNSS 基线与形变曲线
- 展示告警与系统状态
- 执行管理与联动操作

## 3. 当前数据库在闭环中的位置

### PostgreSQL

放业务事实：

- 设备
- 站点
- 规则
- 告警
- 权限
- 审计

### ClickHouse

放高频遥测历史：

- 原始或标准化稀疏点位
- 供历史曲线与聚合查询使用

### Redis

放缓存与运行态辅助能力：

- 去重
- 计数
- 限流
- 临时状态

## 4. 当前闭环中已经明确的部分

- 技术链路已经明确
- MQTT 契约已经明确
- ClickHouse 稀疏模型已经明确
- Desk 需要的主要接口已经列出来

## 5. 当前闭环中最可能断开的部分

### 断点 A：设备侧与平台侧

- 设备上报协议是否与当前平台完全一致
- GNSS 与多传感器字段是否已统一

### 断点 B：ingest / writer 实现

- 当前主仓中的服务源码是否完整
- 是否还能按当前文档恢复运行

### 断点 C：API 与 Desk

- Desk 当前仍使用部分 legacy `/api/*`
- 平台主设计是规范化的 v2 契约

### 断点 D：算法与接口输出

- 形变、基线、风险分析算法的输入输出是否和平台字段统一

## 6. 当前最小闭环建议

先不要一次打所有传感器，建议先打通一条最小链：

- 一台设备
- 一类核心 GNSS / 位移数据
- 一条标准遥测上报
- 一次 ClickHouse 入库
- 一个 API 曲线查询
- 一个 Desk 页面成功展示

## 7. 最小闭环的验收标准

- 能模拟或真实发送一条标准遥测
- Kafka 中能看到消息
- ClickHouse 中能查到数据
- API 能查到结果
- Desk 能切到 HTTP 模式并显示真实数据

## 8. 这份文档的用途

以后任何新工作都要回答：

- 它属于闭环中的哪一段
- 它打通的是哪一段
- 它会不会影响上下游字段与契约
