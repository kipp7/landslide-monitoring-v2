# 后续动作统一建议

## 1. 当前阶段的总策略

先恢复基线，再验证闭环，再补硬件与算法，最后进入稳定并发开发。

## 2. 推荐执行顺序

### 阶段 A：恢复基线

目标：

- 确认当前主仓中的平台链路还能否恢复
- 确认桌面端当前联调状态

建议动作：

- 核查 `infra/compose`
- 核查 `services/api`
- 核查 `services/ingest`
- 核查 `services/telemetry-writer`
- 核查 Desk 当前 HTTP 接口落点

### 阶段 B：打通最小闭环

目标：

- 让一条最小数据链从上报到展示完整跑通

建议动作：

- 用脚本或模拟设备发送一条标准遥测
- 核查 MQTT → Kafka → ClickHouse
- 核查 API 查询
- 核查 Desk HTTP 模式展示

### 阶段 C：接口统一

目标：

- 把 Desk 需要的接口与平台 API 做一致化

建议动作：

- 建一份 Desk ↔ 平台 API 对齐表
- 标出已实现、缺失、字段不一致、待迁移项
- 优先统一：设备、监测站、GNSS 基线、GPS 形变、系统状态

### 阶段 D：GNSS / 硬件 / 算法收口

目标：

- 把历史资料提炼成当前执行版规范

建议动作：

- 整理 GNSS 协议与模块说明
- 整理设备身份、Topic、命令回执
- 整理基线、位移、趋势、风险算法卡片
- 建立固件正式入口

### 阶段 E：进入受控并发开发

目标：

- 让后续每条 Codex 线都有明确边界

建议动作：

- 硬件线
- 平台链路线
- Desk 联调线
- 算法线

每条线都先建任务单再开工。

## 3. 当前最优先的三件事

### 第一优先级：服务与基础设施核查

这是后续一切的前提。

### 第二优先级：Desk ↔ API 对齐

这是当前桌面端从“演示可用”走向“真实联调可用”的关键。

### 第三优先级：GNSS / 基线 / 形变资料收口

这是当前核心业务能力的知识中心。

## 4. 不建议现在做的事

- 不建议继续大范围新增 UI 页面
- 不建议先深挖所有单片机细节
- 不建议同时改协议、服务、客户端、算法且不留任务单
- 不建议继续让多个仓库都成为开发入口

## 5. 推荐的下一轮任务拆分

### 任务 1：平台恢复核查

- 输出：服务现状清单、可运行性结论、缺口清单

### 任务 2：Desk API 对齐清单

- 输出：接口对齐表、优先修复顺序

### 任务 3：GNSS 与基线资料收口

- 输出：当前执行版 GNSS / 基线 / 形变资料清单

### 任务 4：固件与设备协议收口

- 输出：设备协议摘要、固件入口建议

## 6. 当前建议的协作方式

- 主开发仓继续作为唯一写入主线
- 参考仓只读
- 每次开始任务先看 `docs/unified/`
- 每次完成阶段性工作后补 `docs/journal/`

## 7. 本文件的作用

本文件不是长期路线图，而是当前阶段的执行顺序确认。

后续若主线变化，应新建下一版统一动作文档，而不是直接覆盖本文件。
