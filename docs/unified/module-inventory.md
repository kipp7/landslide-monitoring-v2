# 模块与职责清单

## 1. 仓库级别清单

### `landslide-monitoring-v2`

当前主开发与集成仓，承担：

- 文档收口
- 并发开发
- 平台与桌面端的当前联调
- 后续统一主线的落地

### `openharmony`

历史参考仓，承担：

- 架构设计依据
- 平台服务与文档参考
- MQTT / 存储 / API / 规则设计参考
- 历史 GNSS、形变、论文与实现材料参考

### `LAMv2_Desk`

桌面端参考仓，承担：

- Windows 客户端实现参考
- Desk UI 与 WPF 壳参考

## 2. 当前主开发仓目录职责

### `apps/`

- 当前主要关注：
  - `apps/desk`
  - `apps/desk-win`
  - 可能保留的 `apps/web` 历史内容

职责：

- 提供桌面 UI、桌面壳、后续可能的 Web 适配

### `services/`

职责：

- 承接设备消息、转发、存储、规则、告警与 API 服务

理想职责拆分：

- `api`
- `ingest`
- `telemetry-writer`
- `rule-engine`
- `notify`
- 其他命令、回执、存在感知与辅助 worker

### `libs/`

职责：

- 放共享校验、规则、可观测性、后续共享 DTO 与算法公共库

### `infra/`

职责：

- 放单机 Compose、环境配置、运维与恢复脚本

### `data/`

职责：

- 本地基础设施数据目录
- 当前已知包含：PostgreSQL、ClickHouse、Kafka、Redis

### `docs/`

职责：

- 当前主线文档入口
- 统一确认、治理、工作记录、算法、硬件等都应逐步在这里收口

## 3. 平台级模块清单

### 设备与采集层

包含：

- GNSS
- 倾角
- 雨量
- 电源/健康状态
- 其他环境或姿态传感器

职责：

- 采集数据
- 本地整理
- 按协议上传
- 接收命令与回执

### MQTT 接入层

职责：

- 接收设备上报
- 按 Topic 分类
- 做最外层鉴权与 ACL 控制

### ingest 层

职责：

- 校验消息结构
- 补充接收时间等字段
- 写入 Kafka
- 异常消息进入 DLQ

### Kafka 层

职责：

- 削峰
- 解耦设备写链与后端消费链
- 作为可回放缓冲层

### telemetry-writer 层

职责：

- 读取标准化遥测
- 拆为稀疏点位
- 批量写入 ClickHouse

### rule-engine 层

职责：

- 根据规则和算法输出告警事件
- 记录证据链
- 不阻塞主写链

### API 层

职责：

- 向 Web / Desk 提供设备、站点、曲线、基线、告警、系统状态等接口

### 客户端层

职责：

- 提供管理、展示、联动与操作入口

## 4. 桌面端模块清单

### `apps/desk`

职责：

- React UI
- 页面路由
- Mock / HTTP 双模式
- 与平台 API 对接

### `apps/desk-win`

职责：

- WPF 宿主
- WebView2 加载前端页面
- 托盘、窗口、全屏等桌面能力

## 5. 业务能力模块清单

### 设备管理

- 设备信息
- 设备状态
- 设备绑定与站点关系
- 设备控制与命令下发

### 监测站管理

- 站点
- 区域
- 网络层级
- 与设备映射关系

### GNSS / 基线 / 形变

- 基准点管理
- GNSS 数据读取
- 形变曲线
- 位移/速度/趋势分析

### 规则与告警

- 阈值
- 组合条件
- 防抖回差
- 事件化告警

### 系统状态与运维

- 服务健康
- 数据链状态
- 异常消息
- 缓存与消息流

## 6. 当前最需要进一步核查的模块

- `services/api`
- `services/ingest`
- `services/telemetry-writer`
- `services/rule-engine-*`
- Desk 当前 HTTP 接口实现

## 7. 这份清单的用途

后续每次开始任务时，先确定它属于：

- 仓库级问题
- 平台级问题
- 桌面端问题
- 硬件问题
- 算法问题

然后再进入并发拆分与实现。
