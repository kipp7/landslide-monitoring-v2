# 2026 软通赛题技术路线逆向评估

## 1. 逆向评估的出发点

不先问“我们想做什么”，而先反过来问：

- 评委最容易质疑什么
- 工程落地最容易失真什么
- 创业化最容易空心化什么
- 你们现在真正有基础的核心能力是什么

只有先把这四个问题答清楚，技术路线才不会越写越大、越写越虚。

## 2. 如果我是评委，我会先看哪三件事

### 2.1 你们是不是有真实系统基础

这一点你们是有优势的。

当前最值得保留并持续主打的，不是空泛地说“AI 预警”，而是：

- 你们已经有滑坡监测系统架构基础
- 已经有 OpenHarmony / TX-SMART-R / 边缘侧链路基础
- 已经有 GNSS 形变监测、多传感采集、平台展示、设备健康这条主链

这意味着你们讲的是“在已有系统上升级核心技术”，不是“从零想象一个大而全方案”。

### 2.2 你们的核心技术是不是足够聚焦

真正能打的核心技术，建议只保留四条主轴：

- 区域自适应形变位移预测
- 数据驱动预警模型
- 多模态感知融合
- 边缘侧数据链健康监控

这四条是能互相支撑的。

### 2.3 你们是不是有长期壁垒

今年最值得写成壁垒的，不是“用了某个大模型”，而是：

- 区域滑坡数据库
- 数据治理与标签体系
- 模型库与模型匹配
- 设备健康与链路质量数据闭环

这才是能长期积累的东西。

## 3. 必须保留并主打的内容

这些内容建议写进主标题、简介、创新点、答辩主线。

### 3.1 区域自适应位移预测模型

这是你们今年最应该主打的升级点之一。

原因：

- 它比“普通规则预警”更像核心技术
- 它和顾昌彪论文、当前行业趋势高度一致
- 它能自然带出“区域模型库”和“模型匹配”

### 3.2 数据驱动预警模型

必须保留。

因为如果只有位移预测，没有预警决策层，作品会停留在“预测系统”，不够完整。预警模型可以作为第二层，把预测值、多源监测值、背景风险和动态阈值统一成风险分级。

### 3.3 滑坡数据库与数据治理体系

必须单独成段。

原因：

- 这是今年 AI 方向最容易打动评委的“长期价值”
- 也是最适合往创业项目方向延展的资产
- 可以反向证明你们不是做一次性比赛模型

### 3.4 边缘侧数据链健康监控

这一条非常值得保留，而且是你们和很多“只做预警算法”的队伍拉开差距的地方。

它能回答一个非常专业的问题：

如果设备链路坏了，模型再强也没用。你们不仅监测滑坡，还监测监测系统本身的健康。

### 3.5 多模态感知融合

建议保留，但表达要克制。

最合理的写法是：

- GNSS / 倾角 / 振动 / 雨量等数值感知为主
- YOLO 视觉识别为辅

不要写成“视觉是主核心”，否则会被追问视觉样本量、标注量和真实部署价值。

## 4. 建议增加，但要控制力度的内容

### 4.1 区域模型库与模型匹配机制

这条非常值得加，而且会让你们显得更专业。

建议写法：

- 针对不同地区、不同坡体类型、不同诱因机制
- 建立不同模型或参数模板
- 新场景按区域特征进行模型匹配或迁移

这比直接说“一个模型通吃全国”更可信。

### 4.2 动态阈值与误报抑制机制

建议加。

因为评委最担心的就是：

- 误报太多
- 阈值拍脑袋
- 现场根本不敢用

所以你们要明确写：

- 阈值不是人工固定写死
- 会根据区域数据、历史样本、时序趋势动态调整

### 4.3 人工复核与专家闭环

建议加一小段，不需要太大。

原因：

- 完全自动预警很容易被质疑
- 加入“专家复核 / 人工标注 / 运维回写”会让方案更稳

### 4.4 数据版本化和模型版本化

建议加，但只写成工程规范，不要讲得太重。

这是为了体现你们项目是：

- 可迭代
- 可复现
- 可运营

## 5. 可以做，但不建议写成主卖点的内容

### 5.1 YOLO 视觉识别

可以保留，但不要把它写成第一创新点。

因为：

- 视觉识别在滑坡预警里更适合做辅助证据
- 单靠视觉难以承担核心预警责任
- 如果写太重，会被追问数据集、标注规模、夜间和恶劣天气效果

建议定位为：

- 裂缝/坡表异常辅助判识
- 多模态证据补充

### 5.2 遥感/InSAR 融合

可以讲方向，但不要承诺过度落地。

因为：

- 它很加分
- 但真正稳定接入、时效匹配和工程落地成本不低

所以更适合写成：

- 可接入的高价值数据源
- 用于区域建模和长期趋势分析

### 5.3 数字孪生 / 三维大屏

可以有，但不要主打。

评委真正关心的是：

- 预测是不是有效
- 预警是不是靠谱
- 数据是不是可信

不是 3D 页面是不是炫。

## 6. 现在不建议重点加，甚至建议砍掉的内容

### 6.1 大而泛的“大模型”表达

不建议主打“DeepSeek 大模型”这种泛表达。

原因：

- 容易空
- 容易被追问落地方式
- 与滑坡时序预测和预警任务并不天然匹配

更稳的写法是：

- 自主训练位移预测模型
- 自主训练预警模型
- 模型匹配与边缘轻量模型

### 6.2 过多传感器堆料

不建议为了显得复杂而继续无限加传感器。

因为：

- 传感器越多，维护成本越高
- 采样同步和清洗复杂度越高
- 作品会显得“杂而不精”

建议保留最核心的：

- GNSS
- 倾角 / 振动
- 雨量 / 温湿度 / 土壤含水率
- 视觉可选

### 6.3 过多场景同时铺开

不建议一口气主打太多行业场景。

建议主场景锁定为：

- 山体滑坡
- 边坡安全

然后把扩展场景写成：

- 景区边坡
- 公路边坡
- 库区边坡
- 矿山边坡

不要让主叙事跑散。

## 7. 从工程落地角度反推，你们最该补的东西

### 7.1 真实可说的数据来源清单

这是必须补强的。

如果你们后面要答辩，老师和评委很可能会问：

- 数据从哪里来
- 样本多少
- 怎么清洗
- 怎么标注
- 怎么验证

### 7.2 评价指标体系

建议明确至少两组指标：

- 位移预测指标
  - RMSE
  - MAE
  - R²
- 预警分类指标
  - Accuracy
  - Precision
  - Recall
  - F1

再加一组系统指标：

- 链路在线率
- 上传时延
- 丢包率
- 边缘侧告警响应时间

### 7.3 最小闭环演示路线

你们要能讲清楚最小闭环：

监测采集
→ 数据清洗
→ 位移预测
→ 预警判别
→ 平台联动
→ 边缘链路自检

只要这条链清楚，方案就稳。

## 8. 最推荐的最终技术路线结构

建议最终收口为“1 个底座 + 4 个核心模块 + 2 个延展方向”。

### 8.1 一个底座

- 山体滑坡区域化数据底座

### 8.2 四个核心模块

- 多源感知与边缘采集模块
- 形变位移预测模块
- 风险预警判别模块
- 数据链健康监控模块

### 8.3 两个延展方向

- 视觉识别与多模态融合
- 创业化平台与行业复制

## 9. 最终结论

### 9.1 你们现在最值得打的，不是“功能很多”

而是：

- 有系统基础
- 有预测预警升级
- 有数据资产路线
- 有边缘健康监控差异点

### 9.2 你们最需要避免的，不是“做得不够多”

而是：

- 方案太散
- 技术点太花
- AI 表达太虚
- 数据路线没讲透

### 9.3 最稳的方向

把作品明确写成：

一个以 TX-SMART-R 和 OpenHarmony 为载体、以区域数据资产为底座、以位移预测和预警模型为核心、以边缘智检为差异化能力的山体滑坡主动预测预警系统。
