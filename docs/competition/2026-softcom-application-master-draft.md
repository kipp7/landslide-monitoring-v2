# 2026 软通赛题申报主稿母版

## 1. 作品名称

基于TX-SMART-R的山体滑坡智能预测预警与区域自适应多模态智防系统

## 2. 一句话定位

这不是一个只会采集数据的滑坡监测终端，而是一套能够按区域自适应训练、按场景智能匹配、按链路实时智检、按数据持续进化的山体滑坡智能预测预警系统。

## 3. 作品简介

本作品面向人工智能+地质灾害防治场景，针对山体滑坡监测预警中样本稀缺、数据分散、区域差异大导致模型与阈值难迁移，多源数据难统一判别，位移趋势与风险等级难提前识别，以及链路稳定性与数据可信度难保障等痛点，依托 TX-SMART-R 与 OpenHarmony/SwanLinkOS，构建覆盖边缘感知节点、边缘网关、云端分析平台和桌面/Web 可视化的山体滑坡智能防灾系统。作品以位移监测、位移预测和风险预警为核心，在既有地质形变监测、多传感采集、MQTT 上报、边云协同与可视化基础上，引入位移预测模型、预警判别模型、YOLO 视觉辅助识别模型和轻量化数据链健康模型，形成灾害体监测 + 数据链监测协同的多模型智能体系。其中，边缘协同层面向 RK3568 构建自托管边缘网关与智能体原生协同运行机制，具备多节点任务路由、状态记忆、健康摘要生成与本地快速响应能力，并部署轻量化边缘智检机制；同时融合不同地区公开数据、现场监测数据和系统运行数据，构建区域数据库 + 数据治理 + 区域模型库 + 模型匹配 + 持续训练闭环，实现按区域自适应训练、按场景智能匹配、按链路实时智检和按反馈持续优化。作品目标是推动滑坡监测系统从传统规则告警升级为面向人工智能+地质灾害防治、具备区域迁移、多模态融合、边缘智能和持续学习能力的新一代AI主动预测预警平台。

## 3.1 300字申报版

本作品面向人工智能+地质灾害防治场景，针对山体滑坡监测预警中事件样本稀缺、数据源分散且标签不足，不同区域诱因机制和演化特征差异大导致模型与阈值难迁移，多源监测数据难形成统一判据，以及位移趋势、风险等级难提前识别、现场链路稳定性与数据可信度难保障等问题，依托TX-SMART-R与OpenHarmony/SwanLinkOS，构建覆盖边缘感知、边缘网关、云端分析和桌面/Web的山体滑坡智能防灾系统。系统围绕位移监测、位移预测和风险预警，融合多传感采集、MQTT上报、边云协同、YOLO视觉辅助识别和轻量化数据链健康模型，形成“灾害体监测+数据链监测”双对象协同机制，可输出风险等级、预警建议与边缘侧快速响应结果；同时整合公开数据、现场监测数据和运行数据，构建区域数据库、模型库与持续训练闭环，实现按区域自适应训练、按场景智能匹配和按链路实时智检，推动滑坡监测由规则告警升级为AI主动预测预警平台。

## 3.2 我们如何把 AI 用实、用到业务闭环里

本作品中的人工智能不是泛化概念，也不是简单接入通用大模型，而是围绕滑坡监测预警场景落到四类可训练、可部署、可验证的专用模型能力：

- 位移预测模型
  - 输入数据：GNSS 位移、倾角、振动、雨量、温湿度、土壤含水率、地下水位与时间序列特征
  - 输出结果：未来一段时间的位移变化趋势、加速度异常和短时风险信号
  - 业务作用：把“事后告警”前移为“趋势预测”
- 预警判别模型
  - 输入数据：预测位移结果、多源监测指标、区域背景因子、历史灾害样本、动态阈值
  - 输出结果：风险等级、预警建议、误报抑制判断
  - 业务作用：把多个传感器数据统一成可执行的风险结论
- YOLO 视觉辅助识别模型
  - 输入数据：摄像头画面、巡检图像、边坡裂缝与坡表异常样本
  - 输出结果：裂缝扩展、坡表异常、落石风险、现场目标变化等视觉证据
  - 业务作用：为数值监测提供辅助证据，不单独承担核心预警责任
- 轻量化数据链健康模型
  - 输入数据：传感器状态、在线率、上传间隔、延迟、丢包、异常波动、数据缺失
  - 输出结果：链路健康评分、异常类型识别、边缘侧快速告警与自检决策
  - 业务作用：不仅监测滑坡体，也监测监测系统本身是否可信

通过这四类模型与“区域数据库 + 数据治理 + 模型匹配 + 持续训练”机制结合，系统可以逐步形成从数据沉淀、模型训练、边缘部署、风险判别到人工复核回写的持续优化闭环。

## 4. 相比去年获奖作品的升级表达

如果说去年的作品解决的是“看得见、传得回、能报警”，那么今年的作品要重点回答“能不能提前判、能不能跨区域适配、能不能看懂现场、能不能监控自己的数据链是否可靠”。今年的升级不是简单增加几个传感器或页面，而是把作品从“智能监测硬件原型”升级为“具备算法壁垒、数据壁垒和平台壁垒的智能预测预警系统”。

可重点概括为以下十点：

- 从“多源采集、无线传输、平台展示、规则预警”升级为“位移预测、风险预警判别、模型匹配、智能联动”。
- 从“智能终端/微型气象站/竞赛原型”升级为“监测终端 + 区域级预测预警模型 + 多模态感知 + 边缘智检”一体化方案。
- 从规则阈值告警升级为数据驱动的动态预测预警。
- 从单点现场数据升级为“公开数据 + 自有监测数据 + 业务运行数据”的区域训练体系。
- 从算法辅助判断升级为自主训练“位移预测模型 + 预警模型”的核心技术体系。
- 从单一数值感知升级为“GNSS/传感器 + YOLO 视觉识别”的多模态融合。
- 从只监测滑坡体升级为同时监测“灾害体 + 数据链”。
- 从边缘汇聚升级为边缘轻量模型、自检和本地快判。
- 从单点原型升级为面向景区边坡、公路边坡、矿山边坡、库区边坡的区域化复制方案。
- 从“能运行的系统”升级为“能随数据增长不断进化的系统”。

## 5. 滑坡数据库与数据资产主线

今年建议把“数据资产”单独写成创新点，而不是只把模型写成创新点。因为 AI 进入工程化阶段后，真正决定作品长期价值的不是单次模型演示，而是数据资产是否可持续积累、可治理、可迁移、可复用。

本项目建议建立“一个总库 + 五类子库”的滑坡数据库体系：

- 区域滑坡基础数据库
  - 区域编码、地形地貌、地质岩性、土地利用、历史灾害、治理状态
- 多源监测时序数据库
  - GNSS 位移、倾角、振动、雨量、温湿度、土壤含水率、地下水位、时间戳
- 遥感与视觉样本数据库
  - 裂缝图像、坡表异常图像、无人机巡检影像、视频帧、遥感影像、标注框
- 预警事件与人工标注数据库
  - 预警等级、实际事件、专家复核、误报/漏报标签、处置结果
- 设备与数据链健康数据库
  - 在线率、电量、信号强度、上传延迟、丢包率、断点重传、传感器异常

## 6. 数据清洗与治理表达

为了保证模型训练与预警判断的可靠性，本项目将构建标准化数据治理流程，对多源异构滑坡数据进行统一编码、统一时间戳、统一坐标系、统一量纲转换，并针对缺失值、噪声值、异常跳变和设备故障数据进行清洗筛查；同时对位移、降雨、环境、视觉和背景地质数据进行时间窗对齐与空间对齐，建立区域级样本索引体系，并配套完成预警标签、事件真值标签、设备健康标签和视觉识别标签的治理工作。在此基础上，进一步引入数据集版本、标签版本、特征版本和模型版本管理机制，使系统具备持续训练、持续校正和持续演进能力。

## 7. 创业化与产业化方向

本项目后续不仅具备技术研发价值，也具备创业转化潜力。团队计划以山体滑坡监测预警为切入点，逐步构建“感知终端 + 边缘网关 + 数据中台 + 预测预警模型 + 运维服务”的一体化产品体系。与传统监测项目不同，本项目将数据库建设作为核心战略方向，通过沉淀多区域、多模态、多周期的滑坡数据资产，持续提升模型训练能力和场景适应能力，最终形成兼具硬件交付能力、软件平台能力和数据服务能力的行业级解决方案。

其商业化路径可以概括为：

- 监测终端销售
- 边缘网关与部署服务
- 监测平台订阅
- 预警模型服务
- 运维与专家复核服务
- 行业数据服务

## 8. 适合在答辩里强调的三个结论

- 今年 AI 元年下，真正值钱的不是单次模型演示，而是长期可积累的数据资产。
- 你们做的不只是一个滑坡报警装置，而是一个可以持续训练、持续升级的滑坡智能预警平台。
- 这个平台后续既能服务科研，也能服务工程，还具备创业和产业化潜力。
