# 开发日记

## 目标

本目录用于连续记录项目推进过程，避免后续再次出现“思路断裂、仓库分叉、文档失忆”的情况。

## 记录原则

- 每次重要梳理、决策、迁移、冲突处理都要记
- 每次并发任务合并前后都要记
- 每次发现并行 worktree 的有效进展还没同步到主线时，都要先补主线日记
- 每次并行 worktree 开工或收口时，都要先补当前 worktree 自己的当月日记
- 每次明确一份新权威文档或废弃旧资料都要记
- 记录以“事实 + 结论 + 下一步”为主，不写空话
- 每次任务完成后，要把本轮 CLI 最终输出原文补进当月日记，不做改写
- `CLI 最终输出原文` 小节必须与终端最终展示完全一致，不得做摘要或润色

## 建议格式

每次记录建议包含：

- 日期
- 背景
- 做了什么
- 当前结论
- 遗留问题
- 下一步
- `CLI 最终输出原文`

## 标准小节

建议固定使用以下标题顺序：

- `背景`
- `本次做了什么`
- `当前结论`
- `遗留问题`
- `下一步`
- `CLI 最终输出原文`

## 当前日志文件

- `docs/journal/2026-04.md`

## 使用方式

- 日常开发：补在当月日志中
- 阶段性收口：同步更新 `docs/project-landscape.md` 和 `docs/project-status.md`
- 若某条任务线有独立上下文，可在 `docs/concurrent/` 中保留任务单，再在本目录记录最终结论
- 若本轮任务有对用户的最终输出，需在当月日志追加 `CLI 最终输出原文` 小节，内容与终端展示保持一致
- 若当前窗口在并行 worktree 中工作，先更新当前 worktree 的日记；若主线还未同步，再把有效事实补到主线当月日记
