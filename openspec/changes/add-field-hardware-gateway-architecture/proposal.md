## Why

当前主线已经明确了平台主链路、设备身份、MQTT 契约、固件行为约束，但“现场硬件节点 + 网关 + 中心平台”的总体架构仍分散在多份文档里，没有形成统一的权威方案。

这会直接带来高风险：

- 容易把现场节点协议、网关职责、平台契约混在一起，导致边界反复变化
- 容易在尚未冻结现场部署、功耗、信号、冗余策略前就进入实现，后续牵一发而动全身
- 真实硬件联调时，无法明确哪些问题属于节点、网关、现场部署还是平台主链

本变更的目标不是直接做实现，而是先收口一份“硬件-网关-平台一体化架构方案”，把已定真值和待决策项彻底压清楚。

## What Changes

- 新增现场边缘架构能力规范：采集节点、现场网关、中心平台三层职责边界
- 明确“网关适配现有平台契约”是推荐路径，而不是让节点直接耦合平台内部实现
- 固化现场部署原则：分区、覆盖、信号、供电、冗余、运维
- 明确节点与网关的功耗/采样/上报分层策略
- 明确 `install_label` 与 `device_id` 的边界，避免现场标签与平台主身份冲突
- 新增现场轻量遥测规范（Field Telemetry Profile），约束高频/低频字段分层、包长预算与分帧要求
- 评估并纳入 `Ubuntu + RK3568` 作为推荐网关硬件候选的适用范围与约束
- 给出分阶段落地建议：一期只做上行采集闭环，二期再补命令/回执/远程配置
- 增加“软件层面先行调试对接”的分阶段方案：
  - 节点侧协议模拟/调试
  - 网关侧适配/缓存/转发调试
  - 网关到现有平台主链的联调闭环

## Impact

- Affected specs:
  - `field-edge-architecture`（新增）
- Affected docs:
  - 后续将影响 `docs/integrations/firmware/*`
  - 后续将影响 `docs/integrations/iot/*`
  - 后续将影响设备注册、网关接入、部署 runbook 等文档
- Affected code (future work, not in this proposal):
  - 现场网关适配器
  - 设备注册/映射模型
  - 可选的网关健康状态与接入观测链路

## Non-Goals

- 本变更不直接修改 RK2206 固件
- 本变更不直接实现现场网关适配器
- 本变更不直接修改平台数据库主链或 API 主链
- 本变更不在本轮确定最终无线制式/天线型号/电池型号等 BOM 级细节
- 本变更不在本轮直接实现硬件驱动级调试工具，只先冻结软件联调边界与顺序
- 本变更不默认要求在当前项目期内落地长期自研网关；当前阶段优先采用轻量化成熟工具完成联调
